# kill 掉 init 进程重启容器

在容器中使用 `kill` 掉 init 进程（即 pid = 1）的方式重启容器！

用于终止进程的主要有 SIGTERM（15）、SIGKILL（9）两个信号，其中 `kill` 命令如果不指定信号的话，默认发送 SIGTERM 信号给指定的进程。

`kill` 命令的粗略执行流程：

1. 用户态执行 `kill` 命令时会调用内核函数 `sys_kill()`
2. 内核在将信号发送给进程时会先调用 `sig_task_ignored()` 函数判断该信号是否可以被忽略，一旦可以被忽略，信号就不会发送给进程。

判断是否可以被忽略时有多个条件，这里仅列出两个信号相关的主要条件。SIGNAL_UNKILLABLE 标记、是否执行默认操作。

其中 1 号进程在创建时会指定 SIGNAL_UNKILLABLE 标记（其他进程则不会有该标记），而 SIGKILL 信号又是无法被捕获的（所以收到该信号后的行为一定是默认操作），故**对于 1 号进程而言 SIGKILL 信号一定会被忽略的，同理 1 号进程也永远不会响应 SIGSTOP 信号**；而 SIGTERM 是可以被捕获的，也就是说**如果用户对 SIGTERM 的行为进行了修改，那么就可以使用 SIGTERM 信号杀死 1 号进程**！

**init 进程永远不能被 SIGKILL 杀死，但是可以被 SIGTERM 信号杀死。**

注：可以在容器的宿主机上通过 SIGKILL 信号杀掉容器的 1 号进程，而无法通过 SIGTERM 信号杀死。容器的用户可能没有宿主机登陆的权限，所以通常不会在宿主机上操作。

 `sig_task_ignored()` 函数在判断该信号要不要发送给指定进程时，还会判断信号是否与进程在同一个 Namespace。

容器中 init 进程收到 SIGTERM 信号被杀死时，会向其他所有的进程发送 SIGKILL 信号，其他进程会被强制杀死而无法 graceful shutdown。为了让其他进程 graceful shutdown，有时间执行清理文件句柄等操作，可以在容器的 init 进程中对收到的信号进行转发，这样其他进程收到的就是 SIGTERM 信号，就可以捕获后执行自定义的 handler 从而清理资源。

# 僵尸进程

Linux 系统中，每一个进程在退出时都会进入僵尸状态（EXIT_ZOMBIR），如果该进程的父进程不对其进行回收清理，该进程就会成为僵尸进程（使用 `top` 可以看到 Tasks 行末尾有僵尸进程的总数）。

补充：僵尸状态下父进程可以查看子进程的 PID、终止状态（退出码、终止原因，比如是信号终止还是正常退出等）、资源使用信息。如果子进程直接消失，那么父进程没有机会掌握子进程的具体终止情况。一般情况下，程序逻辑可能会依据子进程的终止情况做出进一步处理：比如 Nginx Master 进程获知 Worker 进程异常退出，则重新拉起来一个 Worker 进程。

而残留的僵尸进程同样会消耗系统中的进程数资源，如果实在容器中，最坏情况进程数达到 pids Cgroup 限制的容器最大进程号数，再启动新进程，如运行 `ls` 时，就会看到 Resource temporarily unavailable 的错误消息。

僵尸进程一定需要**其直接的父进程**调用 `wait()` 或者 `waitpid()` 系统调用来清理，如果 init 进程创建了 B、B 又创建了 C，那么当 C 成为僵尸进程，而 B 正常时，init 进程也无法清理掉 C 的，除非 B 也成为僵尸进程，init 进程才可以清理 C。
